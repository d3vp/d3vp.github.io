Python Style Guide

This style guide is adapted from https://courses.cs.washington.edu/courses/cse160/16wi/handouts/style_guide.html for this course.

Naming variables

  • Variable names in Python should be descriptive, concise, and lowercase, with words separated by underscores.
  • Students shouldn’t use single letter names for variables such as i or j except as an index in for loop or when used in some mathematical formulas.
  • In Python, it’s recommended to name variables in a certain way so that other Python programmers can read your code more easily. Here are some examples of variable names: “num_lines”, “running_sum”, “my_friends”, etc.

Good:

1total
2height_sum
3average_weight
4age

Bad:

1t  # Fine as a variable in for loop but not in general
2NumberSum  # use underscore between words
3a_w  # Not clear what this variable is for
4whats_my_age_again  # too verbose, detracts from readability

Space between operators

  • Include spaces between arithmetic, logical and other operators in an expression.
  • Limit to 1 space, to avoid unnecessary whitespace.
  • The exception to this is parentheses, which can be directly adjacent to whatever they are enclosing.

Good:

1x = 4
2x + y
3(sum ** 2) + 4 * val - 1
4x * (4 + 6)
5b + math.sqrt(4 * max_val)

Bad:

1x=4  # not enough spaces
2x+y  # not enough spaces
3(sum**2)+4*val-1  # not enough spaces
4x * ( 4 + 6 )  # unnecessary spaces around parentheses
5b +   math.sqrt( 4 *     max_val)  # inconsistent spacing

Space between function names and their arguments

  • Avoid adding extra space(s) between a function name and its associated arguments.
  • For example, you should write, range(x) instead of range (x) (in other words, do not put a space between ‘range’ and the opening parenthesis). Using the space suggests—incorrectly—that the parenthesis are for grouping an expression, when in fact they are for calling the function.
  • You should, however, include spaces between individual arguments in the argument list. This makes your function calls more readable.

Good:

1length = len(message)
2range_vals = range(n, 4)

Bad:

1length = len (message) # too much space before the parenthesis
2range_vals = range(n,4)  # no spaces between arguments

Blank Lines

  • There should be two blank lines separating function definitions from other portions of code.
  • Try to minimize blank lines within function definitions, except when separating complex chunks of code/logic to provide readability.

Good Example (2 blank lines separating function and other code):

1# other code
2

3

4def compute_avg(x, y, z):
5    # function code block
6

7

8# other code
9print(compute_avg(1, 5, 3))

Good Example (a blank line separating chunks of code in a function):

1def compute_avg(x, y, z):
2    """
3    Sums the given numbers and returns the average.
4    """
5    sum_val = x + y + z
6

7    # compute and return the average value
8    result = sum_val / 3.0
9    return result

Bad Example (not enough blank lines):

1# other code
2def euclidean_distance(x1, y1, x2, y2):
3    # function code block here
4print(euclidean_distance(10, 20, -5, 7))

Bad Example (too many blank lines within a function):

1def euclidean_distance(x1, y1, x2, y2):
2    """
3    Computes Euclidean distance between two 2D points.
4    """
5

6

7    d = (x1 - x2) ** 2 + (y1 - y2) ** 2
8

9

10    return d ** 0.5

Line Length

  • Limit lines of code/comments to 80 (or max 90) characters to provide readability.
  • If a line extends past this limit, split it onto another line.
  • Use parentheses to group an expression so that you can break long line as shown below.

Bad:

1# Line too long, need to scroll to read
2income = gross_wages + taxable_interest + 10.5 * (dividends - qualified_dividends) - ira_deduction - student_loan_interest

Good:

1# break into shorter lines, allowed because of parentheses
2income = (gross_wages + taxable_interest 
3          + 10.5 * (dividends - qualified_dividends) - ira_deduction
4          - student_loan_interest)

Bad:

1# Line too long, need to scroll to read
2print("Value of number is", number, "& square of number is", number ** 2 , "& cube of number is", number ** 3)

Good:

1# break into shorter lines, allowed because of parentheses
2print("Value of number is", number, "& square of number is", number**2,
3      "& cube of number is", number**3)

Commenting

  • Include a header comment at the top of a file to describe what the purpose of a python program is.
  • Also, long blocks of code with a particular purpose or small bits of particularly complex code should be labeled/explained by a comment on the preceding line(s).
  • Avoid commenting on information that is unnecessary or obvious to other programmers reading your code (e.g. comments like “initialize variable” or “using for loop” are not informative).
  • Use proper spacing with your comments as well, one space after hashtag (#)
1#Don't do this (no space after hashtag)
2# Do this (one space after hashtag)
3x = 5# This is too close to the code!
4x = 5                # Too much space!
5x = 5  # This is better
6

7# Long comment like this one should be on their own line
8print(123)  # short comment ok here

Avoid unnecessary nested if statements or a series of if statements

Avoid nesting if statements that can be written as one if statement.

Do this:

1if val > 120:
2    print "Green!"
3elif val > 80:
4    print "Blue!"
5elif val > 32:
6    print "Red!"
7else:
8    print "Orange!"

Instead of this:

1if val > 120:
2    print "Green!"
3else:
4    if val > 80:
5        print "Blue!"
6    else:
7        if val > 32:
8            print "Red!"
9        else:
10            print "Orange!"

If the intent is for only one block of code to execute, be sure to use if, elif, else instead of a series of if‘s.

Do this:

1if color == 'Red':
2    # code here
3elif color == 'Green':
4    # code here
5elif color == 'Blue':
6    # code here
7elif color == 'Yellow':
8    # code here

Instead of this:

1if color == 'Red':
2    # code here
3if color == 'Green':
4    # code here
5if color == 'Blue':
6    # code here
7if color == 'Yellow':
8    # code here

Lingering Debug Code

  • Please remove any print statements or unnecessary computations that you added to help debug errors before you turn in your program.
  • Don’t just leave unnecessary code commented out. It makes the real code harder to read.

Unnecessary Use of Parentheses

Avoid using excessive parentheses in your code, when not needed. Too many parentheses detract from readability.

Good:

1(x + 4) / (6 ** (y ** 2))   # readable

Bad:

1((((x) + (4))) / ((6) ** ((y) ** 2)))   # less readable
